# 轨道交通模拟系统初步目标需求分析

## 一、项目定位

本项目面向轨道交通运行模拟，目标是在有限实训周期内实现一个可演示、可交互、可扩展的轨道交通模拟系统。

系统应能够模拟列车在线路上的基本运行过程，并体现车辆动力学、线路条件、信号约束、供电状态、车门联锁和运行日志等核心要素。

初步阶段不追求完整复刻专业轨道交通仿真软件，而是优先完成一套能稳定运行、便于展示、后续可继续扩展的基础系统。

## 二、核心用户与使用场景

### 2.1 核心用户

| 用户类型 | 主要需求 |
| --- | --- |
| 操作者 / 司机 | 手动控制列车牵引、制动、惰行、开关门 |
| 调试人员 | 观察列车速度、位置、信号、限速、供电状态 |
| 答辩评委 | 看到系统能够体现真实轨道交通运行约束 |
| 项目组成员 | 基于模块化结构继续扩展系统功能 |

### 2.2 典型使用场景

1. 手动驾驶列车进站停车。
2. 自动驾驶列车根据距离和速度自动停站。
3. 列车经过限速区段时，系统自动读取当前限速并进行提示或约束。
4. 列车遇到红灯、供电异常、车门未关等情况时，系统禁止发车或触发制动。
5. 系统记录停车误差、超速次数、安全事件等运行结果。

## 三、功能性需求

## 3.1 车辆仿真模块

车辆仿真模块是系统的核心模块，负责模拟列车的运动状态和控制响应。

### 3.1.1 主要功能

| 功能 | 说明 |
| --- | --- |
| 速度计算 | 根据牵引、制动、阻力、坡度等因素更新速度 |
| 位置计算 | 根据速度随时间推进列车位置 |
| 加速度计算 | 根据当前控制级位和线路条件计算加速度 |
| 牵引控制 | 支持不同牵引级位，影响列车加速 |
| 制动控制 | 支持常用制动和紧急制动 |
| 惰行控制 | 司控器回零后，列车依靠惯性继续运行 |
| 车门状态 | 支持开门、关门和门联锁判断 |
| 运行模式 | 支持手动驾驶和自动运行的基础区分 |

### 3.1.2 基本动力学要求

初步阶段可采用简化动力学模型，但应至少体现以下现象：

- 上坡时列车加速能力下降。
- 下坡时列车制动距离变长。
- 速度越高，空气阻力影响越明显。
- 牵引、制动、惰行三种状态下列车运动表现不同。

## 3.2 轨道线路模块

轨道线路模块负责提供线路基础数据，是车辆运行、限速判断、停车判断和信号约束的基础。

### 3.2.1 主要功能

| 功能 | 说明 |
| --- | --- |
| 站点数据 | 维护站名、站点位置、上下行站台等信息 |
| 区间距离 | 支持计算当前位置到下一站的距离 |
| 限速区段 | 根据当前位置获取当前允许速度 |
| 坡度区段 | 根据当前位置获取当前坡度 |
| 站台侧信息 | 判断到站后允许打开哪一侧车门 |

### 3.2.2 数据使用范围

项目已有 `线路数据(1).xls`，其中包含 33 个 Sheet。初步阶段建议优先使用以下数据：

| 数据表 | 用途 |
| --- | --- |
| 车站表 | 获取车站列表 |
| 站台表 | 获取站台位置和站台方向 |
| Seg 表 | 获取线路拓扑和区段长度 |
| 静态限速表 | 获取线路限速信息 |
| 坡度表 | 获取线路坡度信息 |
| 信号机表 | 获取信号机位置和方向 |

初期可以先将原始 Excel 数据转换为简化 JSON 或程序内部数据结构，保证系统能稳定运行。

## 3.3 信号系统模块

信号系统模块负责模拟列车运行过程中的基本安全约束。

### 3.3.1 主要功能

| 功能 | 说明 |
| --- | --- |
| 信号状态 | 支持红灯、黄灯、绿灯三种基本状态 |
| 红灯防护 | 红灯前必须停车或触发制动 |
| 黄灯限速 | 黄灯状态下降低允许速度 |
| 绿灯放行 | 允许列车正常运行 |
| 发车许可 | 判断列车是否允许从站台发车 |
| 停车防护 | 异常情况下触发紧急制动 |

### 3.3.2 初步信号规则

初步阶段可采用简化信号逻辑：

```text
绿灯：允许运行
黄灯：允许运行，但需要降低速度
红灯：禁止越过，接近时应自动制动
```

后续可继续扩展进路、保护区段、接近区段和触发区段等更完整的联锁逻辑。

## 3.4 供电状态模块

供电状态模块用于模拟供电条件对列车牵引能力的影响。

| 供电状态 | 系统行为 |
| --- | --- |
| 供电正常 | 列车可正常牵引 |
| 电压低 | 限制牵引能力 |
| 断电 | 禁止牵引，只能惰行或制动 |
| 故障恢复 | 恢复正常运行能力 |

初步阶段不需要建立复杂供电网络模型，重点体现供电状态会影响车辆控制。

## 3.5 车门与站台联锁模块

车门与站台联锁模块用于保证列车停车、开门和发车过程符合基本安全规则。

| 约束 | 说明 |
| --- | --- |
| 未停稳不能开门 | 列车速度必须为 0 或接近 0 |
| 车门未关不能发车 | 任意车门打开时禁止牵引 |
| 到站后只能开正确侧门 | 根据站台侧信息判断左侧或右侧开门 |
| 非站台区域禁止开门 | 列车不在站台范围内时不能开门 |

该模块适合在演示中重点展示，因为规则直观，能够体现轨道交通系统的安全联锁特征。

## 3.6 可视化展示模块

可视化展示模块负责将系统运行状态直观展示给用户。

### 3.6.1 展示内容

| 展示项 | 说明 |
| --- | --- |
| 当前速度 | 显示列车实时速度，单位建议为 km/h |
| 当前位置 | 显示列车在线路中的当前位置 |
| 当前站 / 下一站 | 显示列车当前运行区间 |
| 下一站距离 | 辅助用户判断停车时机 |
| 当前限速 | 显示当前位置对应的线路限速 |
| 当前坡度 | 显示当前位置坡度信息 |
| 信号状态 | 显示红灯、黄灯、绿灯 |
| 供电状态 | 显示正常、低压、断电等状态 |
| 车门状态 | 显示左门、右门、关闭状态 |
| 运行日志 | 记录关键操作和系统事件 |

### 3.6.2 初步界面要求

初步阶段界面可以优先实现以下内容：

- 状态面板。
- 控制按钮。
- 线路位置显示。
- 日志列表。
- 停车误差和安全事件统计。

## 3.7 日志与运行评价模块

日志与运行评价模块用于记录系统运行过程和输出运行结果。

### 3.7.1 日志记录内容

| 日志类型 | 说明 |
| --- | --- |
| 发车事件 | 记录发车时间、站点和运行模式 |
| 到站事件 | 记录到站时间和停车误差 |
| 超速事件 | 记录当前位置、当前速度和当前限速 |
| 红灯违规 | 记录是否越过红灯 |
| 紧急制动 | 记录触发原因 |
| 供电异常 | 记录异常类型和恢复时间 |
| 通信异常 | 可作为后续扩展功能 |

### 3.7.2 运行评价指标

| 指标 | 说明 |
| --- | --- |
| 停车误差 | 列车停车位置与目标停车点之间的距离 |
| 最大速度 | 本次运行过程中的最高速度 |
| 超速次数 | 超过当前限速的次数 |
| 安全事件数 | 红灯违规、异常制动、车门违规等事件数量 |
| 通信成功率 | 后续扩展时用于评价通信状态 |

## 四、非功能性需求

| 类型 | 要求 |
| --- | --- |
| 可演示性 | 系统必须能稳定完成一段完整运行演示 |
| 可扩展性 | 后续能够加入天气、载荷、黏着、防空转、防滑等功能 |
| 可维护性 | 车辆、线路、信号、供电、界面等模块边界清晰 |
| 实时性 | 状态刷新应较为流畅，能体现列车连续运动 |
| 容错性 | 数据缺失或异常时不能直接导致系统崩溃 |
| 可解释性 | 日志和界面应说明为什么限速、制动或禁止发车 |

## 五、需求优先级分析

建议采用 MoSCoW 方法划分需求优先级。

| 优先级 | 需求内容 |
| --- | --- |
| Must 必须完成 | 车辆运行、线路站点、限速、坡度、手动控制、状态显示、日志记录 |
| Should 应该完成 | 自动停站、信号约束、供电异常、车门联锁 |
| Could 可以完成 | 天气、载荷、黏着、防空转、防滑 |
| Won't 初期不做 | 完整 CBTC、完整联锁表、真实供电网络、复杂列车编组仿真 |

## 六、初步目标 MVP 定义

初版系统可以定义为：

> 一列车在真实或简化线路数据上运行，用户可以控制牵引、制动、惰行和车门；系统根据当前位置读取限速、坡度、站点和信号状态，并对超速、红灯、供电异常、车门未关等情况进行约束和日志记录。

该 MVP 能够覆盖立项中的基础目标，并为后续扩展动力学优化场景提供基础。

## 七、后续扩展方向

在初步目标完成后，可以继续围绕动力学细节进行深化。

### 7.1 天气与黏着条件

引入晴天、雨天、冰雪等外部环境变量，使用轮轨黏着系数限制最大有效牵引力和制动力。

可扩展功能包括：

- 天气切换。
- 空转报警。
- 滑行报警。
- 自动削弱牵引或制动级位。

### 7.2 载荷与车辆质量

引入空载、定员、超员等载荷模式，使车辆总质量影响加速度和制动距离。

可扩展功能包括：

- AW0 空载模式。
- AW2 定员模式。
- AW3 超员模式。
- 自动停站制动曲线根据车辆质量动态调整。

## 八、总结

本项目初步目标应聚焦于“能运行、能控制、能约束、能展示、能记录”。

车辆仿真模块负责列车如何运动，轨道线路模块负责列车在哪里运行，信号和供电模块负责列车能不能安全运行，可视化和日志模块负责让运行过程可观察、可评价。

在实训时间有限的情况下，应优先保证 MVP 稳定可演示，再逐步扩展天气、载荷、黏着、防滑等更深入的动力学功能。
