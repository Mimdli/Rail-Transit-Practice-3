# 车辆仿真模块开发方案（多体列车动力学）

> 目标：独立完成多体列车动力学仿真模块（逐节计算 + 车钩力），不依赖联锁、供电、可视化等其他模块。
>
> **适用范围**：本方案仅覆盖多体类列车仿真（逐节动力学 + 车钩力模型）。单体质点简化模式不在本文档范围内。

---

## 一、模块边界

### 1.1 输入 / 输出

```
输入:  线路数据 (限速/坡度/隧道) + 控制指令 + 环境参数
输出:  每节车的新位置 + 新速度 + 车钩力 + 力学报告
```

### 1.2 不依赖的外部模块

本模块是系统中**依赖最少的模块**。其他模块依赖车辆状态的输出，但不影响车辆动力学内部计算逻辑。开发期间所有外部依赖均通过 Mock 实现替代。

## 三、分层架构

多体列车动力学采用自底向上的 8 层结构：

```
Layer 8: VehicleController          ← 顶层入口，编排仿真步
         │
Layer 7: PerCarDynamicsPipeline     ← 七步闭环：逐节计算 + 车钩力
         │
Layer 6: CouplerForce               ← 车钩力（弹簧-阻尼-间隙模型）
         │
Layer 5: ForceCalculator (单车)     ← 单车阻力/牵引力/制动力
         │
Layer 4: TrainConsist               ← 编组配置（动车/拖车排列）
         │
Layer 3: CarConfig                  ← 单车物理参数 + 车钩参数
         │
Layer 2: CarState                   ← 单车状态快照
         │
Layer 1: TrackPosition              ← 一维线路位置坐标
```

**依赖规则**：上层只依赖直接下层，不跨层调用。

### 3.1 物理时钟与 UI 时钟解耦（关键架构决策）

**核心问题**：真实火车车钩刚度极大（$K \approx 10^7\,\text{N/m}$），属于**刚性方程（Stiff Equation）**。若直接按 UI 帧率（$\Delta t \approx 0.033\text{s}$，约 30fps）进行单步显式积分，数值解会立即发散，产生**数值爆炸（Numerical Explosion）**：车钩力的微小位移误差被 $K$ 放大 $10^7$ 倍，加速度剧烈震荡，仿真彻底崩溃。

**解决方案——微步长解算器（Sub-stepping Solver）**：在不改变底层弹簧-阻尼-间隙物理模型（不退化为刚体）的前提下，将物理积分步长与 UI 渲染步长完全分离：

```
外部调用（UI / 调度器）:
  Δt_ui = 0.033s (30fps) 或更大
  → 调用 Pipeline.step(dt=0.033)

Pipeline 内部（微步长解算器）:
  Δt_phy = min(Δt_ui, 0.001s)    ← 由数值稳定性条件决定
  → 将 Δt_ui 切分为 N 个微步
  → for i in range(N):
       执行七步闭环（算力 → 车钩力 → 积分）
  → 汇总最终状态返回给外部
```

**设计要点**：
- 外部调用者完全感知不到微步长的存在——接口签名不变，仍传入 `dt`，仍获得状态快照
- 微步长上限 `dt_phy_max = 0.001s` 是硬约束，需在 Pipeline 内部强制保证
- 车钩力计算是唯一需要微步长的物理过程（阻力/牵引力等低频力可在最外层只算一次以优化性能，但**车钩力必须在每个微步内重新计算**）

| **职责** | 计算单节车受到的各种力（纯函数，无状态） |
| **函数清单** | `calc_davis_resistance(car, velocity) → float`（基本阻力）、`calc_grade_resistance(car, gradient) → float`（坡道阻力）、`calc_tunnel_resistance(car, is_tunnel) → float`（隧道附加阻力）、`calc_tractive_force(car, velocity, throttle) → float`（牵引力，含黏着限制）、`calc_brake_force(car, velocity, brake_level) → float`（制动力，含黏着限制）、`calc_total_resistance(car, velocity, gradient, is_tunnel) → float`（总阻力聚合） |

**牵引力曲线关键验证点**：

```
牵引力曲线检查点（用具体数值验算）:
  v = 0 m/s:   F = F_max              (恒力矩区)
  v = 15 m/s:  F = P / v             (恒功率区)
  v = 22 m/s:  F ≈ 0                 (构造速度，自然衰减)

黏着限制验算:
  干燥, m = 200t: F_adhesion_max = 0.18 × 200000 × 9.81 ≈ 353 kN
  输入牵引请求 400kN → 输出应截断为 353kN ✓
```

**阻力手算验算**：

```
输入: speed = 20 m/s, gradient = +30‰, tunnel = True
手算:
  R_davis    = A + B×20 + C×400
  R_gradient = m × g × 0.03
  R_tunnel   = extra_drag（查表/经验公式）
期望: 代码输出与手算误差 < 1%
```

| **测试要求** | 每种力至少 5 个测试用例，与手算结果比对，误差 < 1% |

### 阶段 6 — 车钩力计算（coupler.py）

| 项目 | 内容 |
|------|------|
| **输出文件** | `vehicle/coupler.py` |
| **职责** | 计算相邻两节车之间的车钩力（纯函数，无状态） |
| **模型** | 弹簧-阻尼-间隙模型（分段线性） |
| **函数签名** | `calc_coupler_force(car_i: CarState, car_j: CarState, config: CouplerConfig) -> float` |

**车钩力模型定义**：

```
设:
  Δx = 实际车距 - 名义车距（车长）
  Δv = 相对速度（后方减前方）
  slack = 车钩间隙（正值为可拉伸/压缩的间隙半宽）

分段:
  若 Δx > +slack:  拉伸区 → F = K × (Δx - slack) + D × Δv  (正值)
  若 |Δx| ≤ slack:  自由区 → F = 0
  若 Δx < -slack:  压缩区 → F = K × (Δx + slack) + D × Δv  (负值)
```

**四个必测工况**（配置: K=1e7 N/m, D=1e5 N·s/m, slack=0.02m, 车长 19.5m）：

```
工况1 — 拉伸 (Δx = +0.03m, Δv = 0):
  0.03 > 0.02 → 拉伸区
  F = 1e7 × (0.03 - 0.02) + 0 = 100 kN (正值 = 拉伸)

工况2 — 间隙内 (Δx = +0.01m, Δv = 0):
  |0.01| ≤ 0.02 → 自由区
  F = 0 ✓

工况3 — 压缩 (Δx = -0.04m, Δv = 0):
  -0.04 < -0.02 → 压缩区
  F = 1e7 × (-0.04 + 0.02) = -200 kN (负值 = 压缩)

工况4 — 拉伸+阻尼 (Δx = +0.03m, Δv = +0.5m/s):
  F = 1e7 × 0.01 + 1e5 × 0.5 = 100k + 50k = 150 kN
```

| **测试要求** | 4 个工况全部手算比对通过 |

> **⚠️ 刚性方程警示**：上述模型中车钩刚度 $K \approx 10^7\,\text{N/m}$ 是真实物理参数，**不可调小**。该量级使车钩力方程成为典型的**刚性方程（Stiff Equation）**：当积分步长 $\Delta t > 0.005\text{s}$ 时，显式 Euler 积分无条件发散（数值爆炸）。**本阶段仅负责力计算（纯函数），不涉及积分**；刚性方程的数值稳定性由阶段 7 的微步长解算器（Sub-stepping Solver）保证。在阶段 6 的单元测试中使用静态输入（固定 $\Delta x, \Delta v$）即可验证公式正确性。

### 阶段 7 — PerCarDynamicsPipeline（微步长七步仿真闭环）

| 项目 | 内容 |
|------|------|
| **输出文件** | `vehicle/dynamics_pipeline.py` |
| **职责** | 接收外部步长 `dt`（如 0.033s），内部切分为微步长 `dt_phy`，逐微步执行七步闭环，返回最终状态 |

#### 7.1 微步长解算器设计（Sub-stepping Solver）

**硬约束**：
- 微步长上限 `dt_phy_max = 0.001s`（1ms），由车钩力刚性方程的数值稳定性条件决定
- 外部传入的 `dt` 可能为任意值（0.01s ~ 0.1s），Pipeline 内部必须保证每个微步 ≤ dt_phy_max

**切分逻辑（伪代码）**：

```python
class PerCarDynamicsPipeline:
    DT_PHY_MAX = 0.001  # 微步长上限（秒），硬约束，不可调大

    def step(self, consist, states, dt, track, env, throttle, brake_level):
        """
        执行一个外部仿真步。
        dt: 外部步长（秒），例如 0.033s（30fps）
        """
        # 1. 切分外部步长为微步
        n_substeps = max(1, math.ceil(dt / self.DT_PHY_MAX))
        dt_phy = dt / n_substeps

        # 2. 低频力（阻力、牵引力、制动力）在外部步长尺度上变化缓慢，
        #    可在最外层计算一次以优化性能
        external_forces = self._calc_external_forces(consist, states, track, env, throttle, brake_level)

        # 3. 微步长循环
        for step in range(n_substeps):
            # Step 1 — 查询线路（每节车当前位置的限速/坡度/隧道）
            track_info = [track.query(s.pos) for s in states]

            # Step 2 — 聚合单车外力（复用预计算的外部力 + 最新车钩力）
            # Step 3 — 计算车钩力（每个微步内必须重新计算！）
            coupler_forces = self._calc_all_coupler_forces(consist, states)

            # Step 4 — 聚合单车合力
            net_forces = self._calc_net_forces(external_forces, coupler_forces)

            # Step 5 — 积分更新状态（使用微步长 dt_phy）
            for i, car in enumerate(states):
                a = net_forces[i] / consist[i].mass
                car.velocity += a * dt_phy
                car.position = track.advance_position(car.position, car.velocity * dt_phy)

            # Step 6 — 限速裁剪
            for i, car in enumerate(states):
                limit = track_info[i].speed_limit
                car.velocity = min(car.velocity, limit)

        # 4. 微步循环结束，汇总最终状态
        # Step 7 — 生成报告
        return self._build_force_report(consist, states, external_forces, coupler_forces)
```

#### 7.2 微步长切分验证（关键）

以下参数组合**必须全部测试通过**，确保切分逻辑正确且无数值爆炸：

| 外部 dt (s) | 预期子步数 | 预期 dt_phy (s) | 验证要点 |
|:-----------:|:----------:|:---------------:|----------|
| 0.033 | 33 | ≈ 0.001 | 典型 30fps 场景，标准切分 |
| 0.016 | 16 | ≈ 0.001 | 60fps 场景 |
| 0.100 | 100 | = 0.001 | 大外部步长，边界压力测试 |
| 0.0005 | 1 | = 0.0005 | 外部步长已小于 dt_phy_max，不切分 |
| 0.001 | 1 | = 0.001 | 恰好等于上限，不切分 |

#### 7.3 性能优化说明

- 阻力、牵引力、制动力（`external_forces`）在微步循环外预计算一次即可——它们在 ~1ms 尺度上变化可忽略
- **车钩力（`coupler_forces`）必须在每个微步内重新计算**——车钩力对位移变化极度敏感（$K = 10^7$），相邻微步间各车位移差已改变，必须更新
- 线路查询（限速/坡度/隧道）在每个微步内更新——各车位置在微步间已推进约 0.02m，可能越过 Seg 边界

| **测试要求** | ① 5 种外部 dt 场景全部测试通过，无数值爆炸 ② 运行加速-巡航-制动完整场景（dt=0.033s），最终停车位置误差 < 2% ③ 验证：dt=0.033 与 dt=0.001（无切分）运行相同物理时长，轨迹偏差 < 0.1% |

**完整场景验收标准**：

```
初始条件: v=0, pos=Seg1.offset=0m, throttle=1.0, dt=0.033s
预期轨迹（内部自动切分为 33 微步/外部步）:
  t ≈ 5s:   v ≈ a × 5 (近似匀加速)
  t ≈ 20s:  v 达到巡航速度 ≈ 22.22 m/s (80 km/h)
  最终:     位置接近目标停车点，误差 < 2%
  全程:     速度曲线平滑，无震荡/发散（证明微步长解算器有效抑制刚性方程）
```

### 阶段 9 — VehicleController（顶层控制器）

| 项目 | 内容 |
|------|------|
| **输出文件** | `vehicle/controller.py` |
| **职责** | 顶层入口，编排仿真循环 |

**核心职责**：

```
1. 持有 TrainConsist（编组）+ ITrackQuery（线路）+ IEnvironmentQuery（环境）
2. 每步调用 PerCarDynamicsPipeline.step()
3. 接收控制指令（throttle, brake_level, target_speed）
```

## 九、开发原则（给实现 Agent 的指引）

1. **自底向上**：严格按阶段 1→9 顺序开发，每阶段产出独立可测
2. **纯函数优先**：力计算函数（forces.py, coupler.py）必须是纯函数——输入参数，返回数值，不持有状态，不访问全局变量
3. **接口隔离**：所有外部依赖通过 ITrackQuery / IEnvironmentQuery 接口访问，不直接依赖具体实现
4. **手算验证**：每个计算函数用固定输入手算预期输出，代码结果与手算比对，误差 < 1% 才算通过
5. **Mock 驱动测试**：开发全过程使用 MockTrackQuery + MockEnvironment，不依赖真实线路数据
6. **集成零改动**：从 Mock 切换到真实实现时，车辆模块内部计算逻辑不需要任何修改
7. **<span style="color:red">⚠️ 微步长数值稳定性测试（必检项）</span>**：Pipeline 测试必须包含专门的数值稳定性验证——用大步长输入（如 dt=0.033s 和 dt=0.1s）运行完整场景，确认：① 微步切分数量正确（`n_substeps = ceil(dt / 0.001)`）；② 速度/位移曲线平滑无震荡；③ 全程无数值爆炸（任一车的加速度不超过物理极限，如 5g ≈ 50 m/s²）；④ 同一物理时长下，dt=0.033s 与 dt=0.001s 的轨迹偏差 < 0.1%。**此项不通过则 Pipeline 不可交付。**
