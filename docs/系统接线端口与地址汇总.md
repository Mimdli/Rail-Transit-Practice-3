# 系统接线、端口与地址汇总

> 更新日期：2026-07-13
> 汇总口径：以 2026-07-13 现场抓包为准；抓包未覆盖的内容再参考 `config/network_config.py` 和接口协议。

## 一、接线总览

各子系统均通过以太网连接，物理接口采用 RJ45、双绞线。总控及接口程序部署机为 `192.168.200.102`，作为各外部系统的数据汇集端。

| 系统 | 接法 | 通信协议 | 数据方向 | 周期 |
|---|---|---|---|---:|
| 供电及车辆仿真 | 模型机与总控机通过以太网互联 | UDP 单播 | 双向 | 20 ms |
| 信号系统 | 信号设备与总控机通过以太网互联 | UDP | 双向 | 信号→总控 250 ms；总控→信号 100 ms |
| 司机台 PLC | 总控机作为 TCP 客户端连接 PLC 服务器 | TCP | 双向 | 100 ms |
| 司机台网络屏 | 总控机作为 TCP 客户端连接网络屏 | TCP | 总控→网络屏 | 100 ms |
| 司机台信号屏 | 总控机作为 TCP 客户端连接信号屏 | TCP | 总控→信号屏 | 100 ms |
| 视景系统 | 总控发送端与本机视景接收端通过以太网互联 | UDP | 总控→本机视景接收端 | 约 100 ms |
| Web ATS | 浏览器连接 FastAPI 服务 | HTTP / WebSocket | 双向 | WebSocket 快照 100 ms |

## 二、当前代码实际使用的地址与端口

| 系统/端点 | 本地地址与端口 | 对端地址与端口 | 角色及说明 |
|---|---|---|---|
| 车辆 UDP | `192.168.200.110:23001` | `192.168.200.102:23002` | 车辆模型绑定本地端口，向总控平台发送并在同一端口接收 |
| 信号网关 | `0.0.0.0:10000` | `192.168.200.106:10000` | 抓包确认信号设备向总控 10000 端口发送 |
| 司机台 PLC | 客户端临时端口 | `192.168.100.123:8001` | PLC 为服务器；8001 为主连接 |
| 司机台 PLC 备用 1 | 客户端临时端口 | `192.168.100.123:8002` | 8001 失败后依次尝试 |
| 司机台 PLC 备用 2 | 客户端临时端口 | `192.168.100.123:8003` | 8002 失败后尝试 |
| 司机台网络屏 | 客户端临时端口 | `192.168.100.121:8888` | 抓包中单帧 570 字节，网络屏为 TCP 服务器 |
| 司机台信号屏 | 客户端临时端口 | `192.168.100.122:9999` | 抓包中单帧 68 字节，信号屏为 TCP 服务器 |
| 视景 UDP | `192.168.100.67:8303` | `192.168.100.124:8303` | 联调电脑绑定 `.67:8303`，周期向视景控制机 `.124:8303` 发送报文 |
| Web ATS（启动脚本） | `127.0.0.1:8000` | 浏览器 | 执行 `python run_web.py` 时使用，仅本机可访问 |
| Web ATS（README 命令） | `127.0.0.1:8765` 或 `0.0.0.0:8765` | 浏览器 | 手动执行 Uvicorn 命令时使用；`0.0.0.0` 可供局域网访问 |

说明：

- “客户端临时端口”由操作系统自动分配，不需要在设备端固定配置。
- 当前 `MODE` 为 `local`，但各地址常量仍为实机地址；只有用户启动“联调连接”后网络模块才会启动。
- 车辆 UDP 如果无法绑定 `192.168.200.110:23001`，代码会退回 `0.0.0.0` 的随机端口。信号网关绑定 10000 失败时也会退回随机端口。退回后虽然可能发送成功，但对端按固定端口回包时将无法正常接收。

抓包确认的单帧载荷长度：

| 数据方向 | 单帧载荷 |
|---|---:|
| 车辆模型 `.110:23001` → 平台 `.102:23002` | 488 字节 |
| 平台 `.102:23002` → 车辆模型 `.110:23001` | 320 字节 |
| PLC `.123:8001` → 上位机 | 46 字节 |
| 上位机 → PLC `.123:8001` | 28 字节 |
| 上位机 → 网络屏 `.121:8888` | 570 字节 |
| 上位机 → 信号屏 `.122:9999` | 68 字节 |
| 联调电脑 `.67:8303` → 视景接收端 `.124:8303` | 154 字节（92 个信号机、40 个道岔） |

TCP 可能把多个应用帧合并在一次接收中，例如抓包中的 56 字节是两个 PLC 28 字节帧，1140 字节是两个网络屏 570 字节帧。

## 四、推荐的现场接线与网卡配置

总控/接口程序需要跨越多个网段。现场可使用多网卡分别接入，也可由支持 VLAN/路由的交换网络承载；不要仅给一张网卡配置 `192.168.200.102` 后就假定能直接访问其他网段。

| 网段 | 主要设备 | 总控侧建议 |
|---|---|---|
| `192.168.200.0/24` | 总控 `.102`、信号 `.106`、车辆模型 `.110` | 总控网卡配置 `192.168.200.102/24` |
| `192.168.100.0/24` | 网络屏 `.121`、信号屏 `.122`、PLC `.123`、视景接收端 `.124` | 增加该网段地址或配置明确路由 |
| `9.38.1.0/24` | 信号链路 2、3 | 使用专用网卡/专网或配置明确路由 |
| `18.32.115.0/24` | 旧协议中的 ATS `.27`、视景控制 `.28` | 现场抓包未使用；仅在恢复旧网络方案时配置 |

交换机侧使用普通以太网直通线连接即可；若设备和电脑直接点对点连接，现代网卡通常支持自动翻转，但现场仍建议使用交换机，便于抓包和排障。

## 六、已发现的口径差异与联调确认项

1. **PLC 协议原文内部矛盾**：摘要写上位机下发 26 字节，但抓包为 28 字节，且数据区包含开关量和车辆速度两个 WORD；代码已按抓包修正为 28 字节。
2. **车辆上行协议少 8 字节**：协议按 20×3 个 double 计算为 480 字节，抓包固定为 488 字节；代码已增加一个置零的尾部保留 double，字段语义仍需设备方确认。
3. **视景旧协议地址已停用**：协议中的 `.27:8302 → .28:8303` 与现场不同，当前联调以 `.67:8303 → .124:8303` 为准。
4. **视景设备数量已变化**：旧协议写信号机 77、道岔 29，抓包为信号机 92、道岔 40。
5. **信号链路 2、3 尚未接入代码**：抓包确认 `9.38.1.2/242`、`9.38.1.3/243` 的 50001 流量真实存在，当前 `SignalGateway` 仍只处理 `192.168.200.106:10000`。
6. **Web ATS 端口仍有两种启动口径**：`run_web.py` 使用 8000，README 使用 8765；二者都能运行，但防火墙应按实际启动方式配置。

## 七、快速连通性检查

项目根目录提供：

- `ping_all.ps1`：依次检查所有已知设备 IP 的 ICMP 连通性。
- `ping_all.bat`：双击后调用上述 PowerShell 脚本。

Ping 只能证明 IP 层基本可达，不能证明 UDP/TCP 服务端口已开放。端口联调还应结合 `Test-NetConnection`、抓包工具及程序的收发计数确认。
