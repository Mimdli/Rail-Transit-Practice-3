## 一、当前结论

现在的调度展示层已基本可用，但操作流程和后台状态机还没有真正闭环。已确认：

- 2 个 P0 流程/安全问题。
- 3 个 P1 主要操作问题。
- 通过代码可以创建上行车，但界面无法设置方向。
- 车辆表面能够运行和到站，但车辆实际 Seg 路径可能与调度锁闭进路不一致。

## 二、必须首先修复的问题

### P0-1：车辆实际路径与调度进路分裂

数据库实测下行 `GGZ → FSP`：

- 调度锁闭进路：`13 → 14 → 17 → 44 → 43 → 45 → 46 → 48 → 49 → 50 → 51`
- 车辆实际路径：`13 → 14 → 15 → 16 → 19 → 20 → 22 → 23 → 235`
- 自动驾驶目标 Seg：`235`
- 正确下行站台 Seg：`51`

上行也存在同类问题：调度进路终点为 `204`，自动驾驶目标却落在 `195`。

#### 根因

`DispatchManager._prepare_next_leg()` 使用两套不同逻辑：

1. 自动驾驶目标使用 `from_absolute(target_station.position)`，在同里程多 Seg 时会选错线路。
2. 调度保留进路使用 `compute_station_route()`，按方向站台计算正确拓扑。
3. `AutoDriveController` 又会独立算一次活动进路，最终和调度进路不一致。

#### 修复方案

- 目标停车点必须通过 `resolve_station_track_position(track, station, direction)` 获取方向站台 Seg。
- `reserved_segments` 作为调度、信号、车辆适配器和自动驾驶的唯一进路源。
- 用 `reserved_segments` 构造 `Route`，并设置到 `TrackDataAdapter.set_active_route()`。
- 禁止 `AutoDriveController` 在调度模式下再独立按绝对里程算路。

#### 验收标准

- 调度保留 Seg、TrackAdapter 活动进路和车辆实际经过 Seg 完全一致。
- 到站时头车位于正确方向站台 Seg，不只是绝对里程接近。
- 下行 GGZ→FSP 应以 Seg 51 为目标。
- 上行对应站间应以上行站台 Seg 为目标。

### P0-2：发车可以绕过扣车和紧急停车

当前 `depart()` 会直接执行：

```python
runtime.held = False
runtime.emergency = False
```

导致：

- 扣车后点击“发车”即可解除扣车。
- 紧急停车后点击“发车”即可清除紧急状态。
- 运行中重复点击“发车”仍返回成功。
- “解除扣车”会立即调用 `depart()`，而不是回到待发。

#### 建议状态转换表

| 当前状态 | 允许命令 | 目标状态 |
|---|---|---|
| 待发 | 设置交路、发车、紧急停车 | 运行/紧急停车 |
| 运行 | 扣车、紧急停车 | 扣车/紧急停车 |
| 扣车 | 解除扣车、紧急停车 | 待发/紧急停车 |
| 紧急停车 | 解除紧急状态 | 待发 |
| 停站/折返 | 紧急停车 | 紧急停车 |
| 进路等待 | 扣车、紧急停车 | 扣车/紧急停车 |
| 交路完成 | 删车、重新设置交路 | — |

#### 状态约束

- `depart()` 只允许从待发状态进入运行。
- `depart()` 不得修改 `held` 和 `emergency`。
- `release()` 只解除扣车并回到待发，不自动发车。
- `restore()` 只允许紧急状态使用。
- `restore()` 必须检查列车零速、停车确认和前方安全条件。
- 重复命令应返回明确失败信息，不应重复记录成功日志。

## 三、主要操作问题

### P1-1：界面没有上/下行选择

当前加车界面只有：

- 列车编号。
- 加入位置。

`DispatchView._add_train()` 没有传入 `direction`，所有界面新增列车默认为下行。

#### 修复方案

- 在“列车配置”区增加“运行方向：下行/上行”。
- 站点选择后显示对应站台和 Seg，例如：`KYL · 下行站台 · Seg 69`。
- 终点站可根据交路推荐方向，但必须在界面明示。
- 加车时调用：

```python
self.manager.add_train(train_id, station_id, direction)
```

### P1-2：加车后仍选中旧列车

已实际复现：

1. 当前选中1车。
2. 添加T2。
3. 表格仍然选中1车。
4. 继续点击“设置交路”。
5. 交路被下达给1车，T2仍未设置。

#### 修复方案

- 加车成功后自动选中新列车。
- 命令区顶部常驻显示：`T2 · 上行 · 待发`。
- 设置交路、发车和紧急操作反馈中都重复车号。
- 选中调度表某车时，同步主窗口受控列车；或明确区分“当前调度对象”和“当前驾驶对象”。

### P1-3：命令按钮重叠且没有状态约束

在程序最低窗口尺寸 `1360×820` 下：

- 发车/扣车。
- 解除扣车/紧急停车。
- 解除紧急状态。

三行按钮会发生纵向重叠。

#### 修复方案

- 根据状态启用/禁用命令按钮。
- 当前合法主操作使用主按钮，危险操作单独保留红色。
- 禁用按钮提供 Tooltip，例如：“列车处于紧急停车，请先恢复状态”。
- 降低态势图固定高度，或让左侧操作区可滚动。
- “列车编组”改名为“列车配置”或“列车管理”，避免与 M/T 编组混淆。

## 四、建议的标准操作流程

### 1. 添加列车

1. 输入车号。
2. 选择上行/下行。
3. 选择加入车站。
4. 界面显示对应方向站台和 Seg。
5. 检查站台区段是否占用。
6. 加车成功后自动选中新车。

### 2. 设置交路

1. 明确显示当前调度对象。
2. 选择运营交路或起终站。
3. 按列车方向计算第一目标站。
4. 展示“下一站＋进路 Seg＋预计方向”。
5. 设置后进入待发状态。

### 3. 发车

1. 检查状态必须是待发。
2. 检查已设置交路。
3. 检查车门关闭。
4. 检查非扣车、非紧急状态。
5. 检查前方进路锁闭成功。
6. 车辆、调度、信号使用同一进路。
7. 状态变为运行。

### 4. 扣车与恢复

1. 扣车后状态为扣车，禁止发车。
2. 解除扣车只回到待发。
3. 需要再次点击发车才能运行。

### 5. 紧急停车与恢复

1. 紧急停车后锁定紧急状态。
2. 发车、解除扣车等按钮禁用。
3. 列车完全停稳后才能解除紧急状态。
4. 解除后回到待发，不自动发车。
